# ForHome Requirements

## Goal

ForHome records family chores, proof, reviewer confirmation, fatigue, badges, daily summaries, tomorrow plans, and family messages in a Firebase-hosted web app. Desktop and mobile browsers should use the same Firebase Auth and Cloud Firestore-backed data without requiring the user's PC to run as the production server.

## Requirements

1. Deploy the app through Firebase Hosting for project `forhome-19317`.
2. Store shared application state in Cloud Firestore at `families/forhome/state/app`.
3. Require Firebase Auth for shared data access.
4. Keep the local PowerShell server as a static test server only.
5. Back up Firestore state to GitHub with dated JSON and Markdown files.
6. Keep future guide documents in `docs`.
7. Keep Codex implementation session notes in `docs/session`, ordered by creation date.
8. Add verification test cases for implemented behavior.

## Functional Scope

- Family member login through Firebase Auth
- Admin settings
- Chore completion records with pending reviewer confirmation
- Parent cross-approval: mom's work is approved by dad, dad's work is approved by mom
- Child-category work by the son requires both mom and dad approval
- Photo proof and memo attachment for completed chores
- Reviewer approval or rejection before work is counted
- Morning/evening child-care assignment between mom and dad
- Child-care time records with start and end times
- Morning 07:00 briefing with yesterday's work and today's approved/requested tasks
- Daily view that shows who did more confirmed work
- End-of-day summary of who did which chores
- Tomorrow plan sharing between family members
- Weekly fatigue and XP summary based on confirmed chores
- Badge history
- Family messages
- Review queue for pending completion checks
- Daily summary generation
- GitHub Actions backup/report export

## Firebase Interfaces

Main Firestore document:

```text
families/forhome/state/app
```

The v1 document stores the same state shape used by the browser app:

- `members`
- `accounts`
- `chores`
- `history`
- `tomorrowPlans`
- `messages`
- `badgeHistory`
- `careAssignments`
- `careSessions`
- `settings`
- `version`
- `updatedAt`

History entries should include:

- `verificationStatus`: `pending`, `approved`, or `rejected`
- `reviewerId`
- `reviewedBy`
- `reviewedAt`
- `reviewNote`
- `proofImage`
- `proofImageName`
- `proofCaption`
- `proofAnalysis`
- `approvalRequests`: array of reviewer decisions with `reviewerId`, `status`, `reviewedAt`, and `reviewNote`

Tomorrow plans should include:

- `fromId`
- `toId`
- `choreId`
- `title`
- `note`
- `targetDate`
- `status`
- `requestStatus`: `pending`, `accepted`, or `declined`
- `declineReason`
- `respondedAt`
- `createdAt`

Care assignments should include:

- `date`
- `morningId`
- `eveningId`
- `updatedAt`

Care sessions should include:

- `date`
- `memberId`
- `startTime`
- `endTime`
- `minutes`
- `note`
- `createdAt`

Default Firebase Auth users:

- `admin@forhome.local`
- `mom@forhome.local`
- `dad@forhome.local`
- `son@forhome.local`

## GitHub Backup

The scheduled workflow writes:

- `data/backups/YYYY-MM-DD/state.json`
- `reports/daily/YYYY-MM-DD.md`

The browser app must not store a GitHub Personal Access Token. GitHub writes are performed by GitHub Actions using repository permissions.

## Verification

- PowerShell scripts must parse successfully.
- The local server must not initialize PostgreSQL for production data.
- The browser app must not depend on `/api/state`.
- Firebase config, Firebase Hosting config, and Firestore rules must exist.
- GitHub Actions backup workflow and backup dry-run fixture must exist.
- The client must expose task proof upload, verification status, reviewer approval, and tomorrow plan storage.
- Backup reports must include reviewer status, proof summary, and tomorrow plans.
- With Node/npm/Playwright installed, desktop, mobile, shared-state mock, proof, and tomorrow-plan flows must pass.
